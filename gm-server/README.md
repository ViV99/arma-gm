# GM Server

The Python brain of the Game Master. Receives game state from Arma, builds LLM context, calls Ollama, validates and returns commands.

## What It Does

1. Receives `POST /api/v1/tick` with current game state (units, contacts, objectives, events)
2. Updates in-memory state (unit positions, order tracking, casualties)
3. Builds an adaptive tactical briefing for the LLM (strategic/tactical/local graph views)
4. Calls the local LLM (Ollama) and parses the JSON response
5. Validates commands (unit exists? node exists? anti-thrashing?)
6. Returns validated commands as JSON

## Prerequisites

- Python 3.11+
- [Ollama](https://ollama.ai) running locally with a model pulled

```bash
ollama pull qwen2.5:14b   # or mistral:13b, llama3:13b, qwen2.5:32b
ollama serve               # starts on http://localhost:11434
```

## Installation

```bash
cd gm-server
python -m venv .venv
source .venv/bin/activate          # Windows: .venv\Scripts\activate
pip install -e .
```

## Running

```bash
# With defaults (reads config.yaml)
python -m gm_server.main

# Custom config file
python -m gm_server.main --config /path/to/config.yaml

# Override host/port
python -m gm_server.main --host 0.0.0.0 --port 8080
```

The server logs startup info and each tick to stdout.

## Configuration

Edit `gm-server/config.yaml`:

```yaml
server:
  host: "127.0.0.1"
  port: 8080

llm:
  base_url: "http://localhost:11434"   # Ollama URL
  model: "qwen2.5:14b"
  temperature: 0.7
  max_tokens: 1024
  timeout_seconds: 30

game:
  max_events_buffer: 50
  max_tick_history: 100
  anti_thrash_ticks: 3   # min ticks before re-ordering a unit

pacing:
  calm_to_buildup: 0.3   # intensity threshold
  buildup_to_peak: 0.6
  peak_to_relax: 0.4
  relax_to_calm: 0.2
  peak_max_ticks: 20

maps_dir: "../shared/maps"
prompts_dir: "src/gm_server/prompts"
```

## API

### `POST /api/v1/tick`

Main game tick. Arma sends current state, server returns commands.

**Request** (GameState):
```json
{
  "tick_id": 42,
  "mission_time": 360.5,
  "friendly_forces": [
    {"id": "grp_alpha_1", "type": "infantry_squad", "size": 8,
     "position": "agia_marina_crossroad", "status": "defending",
     "health": 0.85, "ammo": 0.7, "current_task": "hold_position"}
  ],
  "enemy_contacts": [
    {"id": "contact_1", "type": "infantry", "estimated_size": "squad",
     "position": "agia_marina_road_south", "confidence": 0.8, "direction": "approaching"}
  ],
  "objectives": [
    {"id": "obj_crossroad", "status": "contested", "threat_level": "high",
     "graph_node": "agia_marina_crossroad"}
  ],
  "events_since_last_tick": [
    {"type": "unit_damaged", "data": {"unit_id": "grp_alpha_1", "casualties": 1}}
  ],
  "resources": {"reserve_infantry": 2, "reserve_motorized": 1,
                "artillery_available": true, "cas_available": false},
  "graph": {
    "strategic": {}, "tactical": {}, "local": null,
    "node_updates": {"agia_marina_warehouse": {"cover_quality": 0.3}}
  },
  "pacing": {"current_phase": "build_up", "intensity": 0.45, "phase_ticks": 4}
}
```

**Response** (TickResponse):
```json
{
  "tick_id": 42,
  "commands": [
    {"type": "move_squad",
     "params": {"unit": "grp_bravo_1", "to": "agia_marina_road_south", "task": "intercept"},
     "priority": "high",
     "reasoning": "Enemy squad approaching from south, send reserve to intercept"}
  ]
}
```

### `POST /api/v1/directive`

Add a text directive to the LLM prompt. It is injected as a high-priority instruction and expires after `ttl_ticks` ticks.

```bash
curl -X POST http://localhost:8080/api/v1/directive \
  -H "Content-Type: application/json" \
  -d '{"text": "Do not use artillery near the church", "priority": "high", "ttl_ticks": 10}'
```

Priority levels: `low`, `normal`, `high`, `critical`.

### `POST /api/v1/override`

Queue direct commands that bypass the LLM. Validated and executed on the next tick.

```bash
curl -X POST http://localhost:8080/api/v1/override \
  -H "Content-Type: application/json" \
  -d '{"commands": [{"type": "move_squad", "params": {"unit": "grp_alpha_1", "to": "agia_marina_church", "task": "defend"}}]}'
```

### `POST /api/v1/control`

Pause or resume LLM decision-making. While paused, only override commands execute.

```bash
curl -X POST http://localhost:8080/api/v1/control \
  -H "Content-Type: application/json" -d '{"action": "pause"}'

curl -X POST http://localhost:8080/api/v1/control \
  -H "Content-Type: application/json" -d '{"action": "resume"}'
```

### `GET /api/v1/status`

Returns current state summary (units, objectives, pacing, last 10 ticks).

### `GET /ui`

Web operator console — shows forces table, objectives, directive/override forms, and tick log. Auto-refreshes every 3 seconds.

### `GET /docs`

Swagger UI with full API documentation (generated by FastAPI).

## Command Types

The LLM can issue these 17 command types:

| Command | Key Params | Effect |
|---------|-----------|--------|
| `position_squad` | unit, location, task, sector | Defensive hold at node |
| `move_squad` | unit, to, task, speed | Move to node |
| `reinforce` | from_reserve, to, composition | Spawn from reserve |
| `set_ambush` | unit, location, trigger_zone | Stealth hold, engage on contact |
| `artillery_strike` | target_node, rounds | Off-map mortar strike |
| `retreat` | unit, fallback_position | Full-speed move to fallback |
| `set_behaviour` | unit, behaviour, combat_mode | Change AI mode |
| `spawn_group` | composition, location, task | Spawn new group |
| `despawn_group` | unit, reason | Remove group from game |
| `set_fortify` | unit, location | Prone defensive position |
| `set_patrol` | unit, route_nodes | Cyclic patrol route |
| `set_overwatch` | unit, location, watch_sector | Standing overwatch position |
| `request_intel` | area | Request L2 graph for next tick |
| `set_alert_level` | level | Change all groups' behaviour |
| `create_roadblock` | location, unit | Block road with unit + barriers |
| `call_cas` | target_node, type | Air support strike |
| `set_priority` | objective, priority | Change objective threat level |

## Testing with Mock Client

Activate the gm-server venv first (it provides `httpx`):

```bash
source .venv/bin/activate   # Windows: .venv\Scripts\activate

# Interactive mode
python tools/mock_arma_client.py --url http://localhost:8080

# Auto mode: runs 20 ticks with 2 s delay (defaults)
python tools/mock_arma_client.py --url http://localhost:8080 --auto

# Custom ticks and delay
python tools/mock_arma_client.py --url http://localhost:8080 --auto --ticks 30 --delay 5
```

Interactive commands: `Enter` = tick, `d` = directive, `o` = override, `s` = status, `p` = pause, `r` = resume, `q` = quit.


## Key Design Decisions

**Anti-thrashing**: The validator blocks re-ordering a unit within 3 ticks unless a new contact, casualty event, or directive justifies it. This prevents the LLM from flip-flopping orders on every tick.

**Stateless LLM**: The LLM receives a full situational summary on every tick. All state lives in `StateManager`, not the LLM. This makes the system robust to LLM failures and allows easy restart.

**Empty response = safe**: If the LLM times out, returns invalid JSON, or issues zero valid commands, the server returns an empty `TickResponse`. Units hold their current positions (safe default).

**Adaptive context**: The server never sends the full graph. It selects L1 tactical views for active combat zones (max 2 zones, ~300 tokens each) and L2 local detail only when the LLM explicitly requests it via `request_intel`.
